# Claude Code 引き継ぎドキュメント — Prospector RING Layout 実装

このドキュメントは、Claude（chat）でデザインを固めた Prospector ZMK ドングル用のカスタムステータス画面レイアウト「RING」を、Claude Code で実装に進めるための引き継ぎ資料です。

---

## このタスクの背景

ユーザーは Prospector（ZMK ドングル + 1.69インチカラーLCD）用のカスタムステータス画面レイアウトを作りたい。Prospector の `feat/new-status-screens` ブランチ（Zephyr 4.1 対応）には既に4つのレイアウト（RADII を含む）があり、ユーザーはその中の RADII を気に入っているが、独自のデザインを別に作りたい、という出発点。

Claude（chat）と対話しながら、配色・タイポグラフィ・レイアウト・状態遷移・アニメーションを段階的に詰めて、最終仕様書（`prospector-battery-rings-spec.md`）と最終モックアップ画像（`mockup.png`）が完成した。完成したレイアウトの名前は **「RING」** とする。

これから Claude Code で **実装フェーズに入る**。

## 重要な前提：既存レイアウトは変更しない

このタスクのスコープを明確にしておく：

- Prospector に既存する4つのレイアウト（RADII 等）には**一切手を加えない**
- 「RING」を **5つ目の選択肢として追加する** だけ
- 既存ユーザーがビルド時に「RING」を選ばない限り、既存の動作・バイナリサイズ・UI は完全に従来通り
- Kconfig の choice ブロックに新エントリを追加する形で統合する（既存エントリは触らない）

このスコープから外れる修正（既存レイアウトのリファクタリング、共通化、命名変更等）は提案・実施しないこと。やる場合は別タスクとして切り出してユーザーに確認すること。

## 設計の経緯（要約）

### 設計判断の根拠

仕様書だけを読むと「なぜこの選択？」が分からない部分があるので、決定の根拠を残します。

- **配色**：WANA STUDIO のミニマルなパレット（Midnight, Dust, Earth, Ocean, Snow）にオレンジアクセント。背景は Snow tint `#FAFAFA`（純白だと眩しすぎる懸念、ただし実機で調整可能）
- **同心円のリング**：当初は装飾だったが「意味がない円は機能として弱い」というユーザー指摘により、L/R ペリフェラルのバッテリーゲージとして再設計
- **Dongle バッテリー**：3つ目のリングとして組み込む案もあったが、3重円は視覚的に過密なため、ヘッダ右に小さなアイコンとして分離（さらに USB 給電時は非表示）
- **時計**：実装重い割に恩恵が薄いため不採用（Prospector に RTC なし、母艦からの同期実装が必要）
- **MOD/STATE チップ**：当初は矩形チップ（CTRL/SHFT/ALT/GUI の文字表示）だったが、実機でのレイアウト崩れを機に丸型アイコン（C/S/A/G + ⇪/あ）に変更。セクションラベル（LAYER/MODIFIERS/STATE）もすべて削除してミニマル化
- **レイヤー名フォント**：Cormorant Garamond（SIL OFL 1.1）に確定。36pt（3ペリフェラル時は30pt）。LVGLに `lv_font_conv` で変換して組み込む
- **WPM → KEYS**：WPM より「セッションでどれだけ打ったか」のカウントの方がモチベになるという判断
- **アニメーション**：起動時の演出（A: リング描画 / B: レイヤー名フェード）と、レイヤー切替時のフィードバック（D: クロスフェード / F: リング点滅）のみ。バッテリー値変動の補間アニメ（I）は当初採用も「実機で見る機会少ない」ため不採用に変更
- **Modifier の左右区別なし**：HID modifier byte の左右を OR 取って、CTRL チップ1つで左右どちらの Ctrl にも反応する仕様。シンプルで実用的
- **未接続ペリフェラル時の挙動**：リングはトラックのみ／数値はハイフン／ラベルとドットも控えめ色

### 採用しなかったアイデア（参考）

将来の拡張で再検討する場合に：

- BLE 信号強度（RSSI）表示
- 連続稼働時間
- 天気アイコン（母艦からのデータ流し込みが必要）
- 打鍵カウントの永続化（日次/週次/累計）

## 実装に入る前の未決事項と確定事項

仕様書 Section 10 と本ドキュメントで挙げた未決事項のうち、いくつかはユーザーとの対話で決定済みです。Claude Code で実装する際には、確定事項はそのまま採用し、残りの未決事項はユーザーに確認するか合理的なデフォルトで進めてください。

### 確定済み事項（実装方針が決まっているもの）

#### ✅ ペリフェラルバッテリー取得方法
**既存4レイアウト（特にRADII）の実装を踏襲する。** Prospectorの既存レイアウトでは独自の取得・通知メカニズムがすでに整備されているため、新規にイベントを設計する必要はない。RADII の実装を読んで同じパターンで購読すること。

#### ✅ ペリフェラル数のサポート範囲
**1〜3個に対応。** ペリフェラル数はキーボードのKconfigで決まる固定値（既存4レイアウトと同じ取得方法）。リング配置・サイズ・配色・数値配置はペリフェラル数で動的に変動する。詳細は仕様書 Section 5.2 のパラメータ表を参照。

#### ✅ IME状態の取得方法
**既存のRADIIレイアウトに実装済みのIME表示を踏襲する。** RADIIのコードを読み、同じ取得方法・同じ purpose で表示する。新規実装は不要。

#### ✅ キーボード名の取得元
**`CONFIG_ZMK_KEYBOARD_NAME` から取得する。** これはキーボード側の `Kconfig.defconfig` で定義される値で、ZMK 標準の機能。実装：
```c
#ifdef CONFIG_ZMK_KEYBOARD_NAME
    const char *keyboard_name = CONFIG_ZMK_KEYBOARD_NAME;
#else
    const char *keyboard_name = "PROSPECTOR";
#endif
```
何らかの理由で `CONFIG_ZMK_KEYBOARD_NAME` から取得できない場合のフォールバック（B案: Prospector固有のKconfigを新設）はユーザーと相談して決定。

#### ✅ KEYS のインクリメント条件
**セッション単位（電源ON以来のカウント）、modifier は含めない（"打鍵" の語感に合わせる）。** ドングル再起動時にリセットされて問題なし。不揮発化は不要。

### 未決事項（実装時に判断）

#### 1. 既存 Prospector コードの構造把握 ⚠️必須

`feat/new-status-screens` ブランチを実際にチェックアウトして、既存4レイアウトすべてのコードを読む必要があります。確認ポイント：

- レイアウトファイルの場所（`boards/shields/prospector_*/widgets/` あたり）
- レイアウト切替の仕組み（Kconfig の choice か？ランタイムか？）
- 既存レイアウトとの責務分離（電源管理・接続管理は共通モジュールか？）
- 命名規約・コード規約（既存コードに合わせる）
- **特にRADIIの実装：ペリフェラルバッテリー購読方法、IME状態取得方法**（上記の「確定済み事項」を実装するため）

これを読まずに書き始めると、既存パターンと違うコードを書いてしまうリスクがあります。

#### 2. レイヤー名の取得元

- 全レイヤーに `display-name` プロパティを必須にするか
- 未設定時のフォールバック（`L0`, `L1`, ...）
- 文字数制限・はみ出し時の挙動（truncate / フォントサイズ自動縮小）

仕様書では Layer 名のフォントサイズは 34px 固定（3ペリフェラル時は 30px）だが、長い名前（"Function" 8文字 = 約 130px 幅）まではギリギリ収まる。それ以上は別途検討。

#### 3. セリフ書体の選定と組み込み

仕様書の Section 4.2 で候補は挙げているが、実装時に1つに絞る必要：

- **Cormorant Garamond** — 上品で線が細い
- **Playfair Display** — モダンで存在感あり
- **Source Serif Pro** — 可読性高め
- **Noto Serif JP** — 日本語対応

選定基準：
- ライセンス（OFL or Apache 2.0 等、再配布可能なもの）
- 必要な文字（ASCII のみ / 日本語含む？）
- ファイルサイズ（日本語フォントは数 MB 〜数十 MB になりフラッシュ容量に影響）
- LVGL 用 C 配列への変換は `lv_font_conv` で

**推奨デフォルト**：Cormorant Garamond または Source Serif Pro、サイズ 30 と 34 の2サイズ用意（ペリフェラル数で切り替え）、ASCII + 基本記号のみ。

#### 7. KEYS のインクリメント条件

仕様書では「modifier 含む / 含まない は実装時に決定」と保留。

- **A. すべてのキー押下でインクリメント**（modifier 含む）
- **B. 非 modifier キーのみインクリメント** ← **推奨**（"打鍵カウント" の語感に合う）
- **C. マクロ展開後 vs 前**

### 重要度：低

#### 4. CAPS と CAPS WORD の区別

仕様書では ZMK の `caps_word` を想定。HID Caps Lock も検知すべきかは要相談。

#### 5. 起動アニメーションのタイミング

ペリフェラル接続確立に数秒かかる場合の挙動：
- リング描画アニメは起動直後 vs 接続後
- 未接続で起動アニメ流した後の再接続時の挙動

#### 6. エラー / Bootloader / Studio 状態の表示

これらは後付けで OK。MVP には不要。

## 推奨実装手順

### Phase 0: 準備

1. **既存4レイアウトの構造を読む**（必須）
   - `git clone` → `git checkout feat/new-status-screens`
   - 既存4レイアウトすべてのソースを読み、命名規約・ファイル配置・コード構造を把握する
   - 特に RADII レイアウトはユーザーが好みと言っているので、見た目だけでなくコード構造も「お手本」にする
   - **RADII の以下の実装方法を抽出**（RING で踏襲する確定事項）：
     - ペリフェラル数の判定方法
     - ペリフェラルバッテリーの取得・購読方法
     - IME状態の取得方法
   - Kconfig の choice ブロックがどこに定義されているか、レイアウトディスパッチ箇所がどこか特定する
2. **セリフ書体を1つ選んで C 配列化**（未決事項3）
   - `lv_font_conv` で変換
   - サイズ30と34の2つを用意（ペリフェラル数で切替）
   - サンプル文字を表示してサイズ感を確認

### Phase 1: スケルトン（最小ビルド）

新レイアウト「RING」のファイルを作成して、まず以下だけが動く状態を作る：

- Kconfig に `ZMK_DISPLAY_STATUS_SCREEN_LAYOUT_RING` を **追加**（既存4エントリは触らない）
- CMakeLists.txt に新規ソースの条件付きコンパイルを **追加**
- レイアウトディスパッチ箇所に新分岐を **追加**
- `ring_status_screen.c` を新規作成し、2ペリフェラル想定で以下を表示：
  - 背景色 `#FAFAFA`
  - ヘッダ左に `CONFIG_ZMK_KEYBOARD_NAME` の値（取れなければ "PROSPECTOR"）
  - 中央に「Base」（Layer 0 を表示、display-name 取得）
  - リング2本（仮の固定値 50% / 50%）
  - 区切り線
  - 右パネルのラベルだけ（チップは枠線だけ、状態反映なし）

ビルド検証：
- `RING` を選択した場合：上記の絵が出る
- 既存4レイアウトのいずれかを選択した場合：従来通り動作（リグレッションがない）

### Phase 2: 基本機能の接続

優先順位順に動的データを接続：

1. **レイヤー名**（`zmk_layer_state_changed` 購読）
2. **ペリフェラルバッテリー**（既存 RADII の購読方法を踏襲）
   - まず2ペリフェラル想定で実装、その後1〜3対応に汎化
3. **D バッテリー**（`zmk_battery_state_changed`）+ USB 検知
4. **Modifier 4種**（`zmk_modifiers_state_changed`）
5. **OUT**（`zmk_endpoint_changed`）

### Phase 3: 拡張機能

6. **CAPS WORD**（caps_word state 購読）
7. **IME**（既存 RADII の実装を踏襲）
8. **KEYS**（`zmk_keycode_state_changed` でカウント、modifier除く）

### Phase 4: ペリフェラル数バリエーション対応

9. **ペリフェラル数 1, 3 への対応**
   - リング数・サイズ・配色・数値配置をペリフェラル数で切り替え
   - レイヤー名フォントサイズも30/34で切り替え
   - 仕様書 Section 5.2 のパラメータ表通りに実装
   - 各バリエーションで実機表示を確認（テストキーボード3種類が用意できるとベスト）

### Phase 5: アニメーション

10. **A: 起動時リング描画**（`lv_anim` で stroke-dasharray 補間）
11. **B: 起動時レイヤー名フェード**（`lv_anim` で opa 補間）
12. **D: レイヤー切替クロスフェード**（layer_state_changed で trigger）
13. **F: レイヤー切替リング点滅**（並行して 150ms タイマー）

### Phase 6: 仕上げ

- 未接続ペリフェラル時の表示（リングはトラックのみ・数値はハイフン）
- レイヤー名の長さ・truncate 対応
- 起動 → 接続確立フローの調整
- 実機でのフォントサイズ・色見え方の調整

## 注意点

### 仕様書のままにするか、調整するか

仕様書の数値（座標・サイズ）は SVG モックアップから決めたものですが、実機の LCD で見ると印象が変わる可能性があります。特に：

- **フォントサイズ**：mockup の見た目だけでなく、実機で読みやすいか
- **チップサイズ 28×22**：mod チップ8個（mod4 + state2 + 余白）が右パネル90px幅にきちんと収まるか
- **リング太さ stroke-width 4**：240px 高の中での視認性

実機表示を確認しながら微調整する前提で。仕様書は「設計意図のスナップショット」であり、聖典ではありません。

### 既存パターンの尊重

Prospector の既存レイアウト（RADII等）と命名規約・コード構造を合わせること。新レイアウトだけ別の流儀で書くと、後でメンテしにくくなります。

### LVGL のバージョン

Zephyr 4.1 に同梱されている LVGL のバージョンを確認。LVGL 8 系と 9 系では API が違います（特に `lv_anim` 周り）。仕様書の Section 7 の LVGL コード例は概念的なもので、実バージョンの API に合わせて書き直す必要あり。

## 実機フィードバック（初回実装後）

初回実装が動作した実機写真を `actual_device_photo.jpg` として同梱している。**機能的には問題なく動作しているが、レイアウト崩れが3点あった。** またこの実機フィードバックを踏まえてデザインも変更している。

![実機写真](./actual_device_photo.jpg)

### 確認された問題と対応方針

#### 問題1: バッテリー残量の数値が折り返されている
ドット + 数値が縦に折り返されている。水平に1行で収まるべき。

**修正方針**: 数値ラベルの折り返しを無効にする（`lv_label_set_long_mode(label, LV_LABEL_LONG_CLIP)` 等）。それでも収まらない場合はドットと数値の間隔を詰めるか、リング中央のスペースを広げる。

#### 問題2: MOD / STATE のチップが右端で見切れている
右パネルの各チップが右端でクリッピングされていた。

**対応（デザイン変更）**: チップを矩形（28×22）から **丸型アイコン（r=12/10）** に変更し、セクションラベル（LAYER/MODIFIERS/STATE）も削除してミニマル化。これにより右パネルの幅問題が根本的に解消された。新しいチップ仕様は仕様書 Section 5.4/5.5 を参照。

#### 問題3: KEYS のラベルと数値が重なっている
「KEYS」テキストと打鍵カウント数値が同じ位置に被っていた。

**修正方針**: 問題2のデザイン変更でチップが小さくなったことで縦方向に余裕が生まれ、自然に解消される見込み。それでも重なる場合は Y 座標を調整する。

### デザイン変更後の確定パラメータ（再実装時に使用すること）

初回実装からの変更点が多いため、以下を必ず確認してから実装すること：

| 変更点 | 旧（初回実装） | 新（確定版） |
|---|---|---|
| MODチップ形状 | rect 28×22, rx=4 | circle r=12 |
| MOD表示文字 | CTRL / SHFT / ALT / GUI | C / S / A / G |
| STATEチップ形状 | rect 28×22, rx=4 | circle r=10 |
| STATE表示文字 | CAPS / IME | ⇪ / あ |
| セクションラベル | LAYER / MODIFIERS / STATE あり | すべて削除 |
| レイヤー名フォント | 未確定 | Cormorant Garamond 36pt |
| MOD配置 | 2×2グリッド（既存） | 2×2グリッド、上端Y=46、r=12、gap=6 |
| STATE配置 | 1行2個（既存） | 1行2個、stR=10、チップ間gap=10 |
| MOD→STATE 間隔 | 未確定 | 14px |

### 修正時の判断基準

- 仕様書の絶対座標（x=206, r=78 等）は**目安**として扱い、実機で見栄えが良いバランスに調整してよい
- ただし**色（HEX値）、フォント（Cormorant Garamond）、チップ形状（丸型）、表示文字（C/S/A/G/⇪/あ）**は仕様書に従うこと

### Output選択画面の仕様

#### レイアウト

- ヘッダなし（Brightness画面と同様、サブ画面ではキーボード名を表示しない）

- 画面最下部にページドット4個（3番目が点灯）

#### ボタン配置

5ボタンすべて同じサイズ。ヘッダなしのため画面上部から使用可能。

- ボタンサイズ：幅 232px × 高さ 32px、rx=8
- 左右マージン：24px（x=24〜256）
- ボタン間隔：8px
- 1個目 y=36、2個目 y=76、3個目 y=116、4個目 y=156、5個目 y=196
- ページドット y=232

```
y=0
y=20  OUTPUT ラベル（中央揃え）
y=36  [ USB              ] ← 32px高

y=116 [ BLE 2            ]
y=156 [ BLE 3            ]
y=196 [ BLE 4            ] ← 下端 y=228

y=240 画面下端
```

#### ボタンの状態

**選択中（現在の出力先）：**
- fill=accent (`#E89B5C`)
- テキスト：ui-sans 12, font-weight=500, color=`#22282C`
- 右端に●インジケータ（r=3, fill=`#22282C`, cx=248）

**非選択：**
- fill=none, stroke=ring.track（Light: `#E2E5E8` / Dark: `#3A4248`）, stroke-width=1.2
- テキスト：ui-sans 12, color=text.secondary（Light: `#5F6A70` / Dark: `#929FA7`）

#### 操作

- 非選択のボタンをタップ → ZMKの `&out` / `&bt` behavior で出力先を即切替
- 切替後、メイン画面に自動遷移（0.3〜0.5秒程度の遅延後）
- メイン画面のOUT表示が新しい出力先に更新される
- 選択中のボタンをタップ → 何もしない（すでにその出力先）

#### ZMK behavior の対応

| ボタン | ZMK behavior |
|---|---|
| USB | `&out OUT_USB` |
| BLE 1 | `&bt BT_SEL 0` + `&out OUT_BLE` |
| BLE 2 | `&bt BT_SEL 1` + `&out OUT_BLE` |
| BLE 3 | `&bt BT_SEL 2` + `&out OUT_BLE` |
| BLE 4 | `&bt BT_SEL 3` + `&out OUT_BLE` |
